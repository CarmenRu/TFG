\apendice{Plan de Proyecto}

\section{Introducción}

En este apéndice se describe la planificación seguida durante el desarrollo del Trabajo Fin de Grado, incluyendo la metodología 
empleada para la organización del proyecto, la planificación temporal, la estimación de los costes asociados a los recursos 
utilizados durante su realización y la viabilidad legal del proyecto.

Para el desarrollo de InteraDD se adoptó una metodología ágil basada en Scrum, organizando el trabajo en diferentes sprints 
con una duración variable en función de las reuniones programadas con las tutoras. Cada sprint se centró en la implementación 
de un conjunto concreto de funcionalidades, lo que permitió desarrollar la aplicación de forma incremental y validar progresivamente 
los resultados obtenidos. Este enfoque facilitó la incorporación de mejoras y la resolución de incidencias conforme avanzaba el 
desarrollo, adaptando el proyecto a las necesidades detectadas durante su implementación.

Como herramienta de control de versiones se utilizó Git junto con GitHub, donde se alojó el código fuente del proyecto y se registró 
la evolución del desarrollo mediante commits. Asimismo, GitHub se empleó para la planificación y el seguimiento del trabajo, 
organizando las tareas correspondientes a cada sprint y permitiendo controlar el estado de las funcionalidades previstas. 
Además, se utilizó GitHub Projects para gestionar el estado de los issues asociados a cada milestone, registrando si se encontraban 
en \textit{backlog}, \textit{in progress} o \textit{done}.

Los apartados siguientes describen con mayor detalle la metodología de desarrollo seguida, la planificación temporal del proyecto, 
la estimación de los recursos económicos necesarios para su realización y la viabilidad legal del proyecto.

\section{Metodología de Ingeniería del Software}
En el desarrollo del proyecto se adoptó una metodología ágil basada en Scrum, adaptada a las necesidades y características específicas del Trabajo Fin de Grado. Scrum es un marco de trabajo orientado a la gestión de proyectos mediante ciclos de desarrollo iterativos e incrementales denominados \textit{sprints}, cuya duración habitual oscila entre una y cuatro semanas. En cada \textit{sprint} se planifica un conjunto de tareas o \textit{issues} que deben completarse antes de iniciar el siguiente ciclo de desarrollo.

En este proyecto, la duración de los \textit{sprints} estuvo condicionada por la planificación de las tutorías. Como norma general, en cada reunión con las tutoras se revisaban los objetivos alcanzados durante el \textit{sprint} anterior, se evaluaba el progreso realizado y se definían las tareas correspondientes al siguiente periodo de trabajo.

La utilización de esta metodología permitió desarrollar la aplicación de forma incremental, facilitando la incorporación de nuevas funcionalidades, la adaptación a los cambios surgidos durante el desarrollo y la mejora continua del algoritmo de análisis de interacciones farmacológicas.

Con el fin de facilitar la comprensión del proceso de desarrollo, este puede dividirse en las siguientes etapas:

\subsection*{Planteamiento inicial y revisión bibliográfica}

La primera fase del proyecto estuvo orientada a la definición de los objetivos generales y de las funcionalidades que debía ofrecer la aplicación. Entre los requisitos funcionales establecidos destacan los siguientes:

\begin{itemize}
    \item Consulta de interacciones entre dos principios activos.
    \item Explicación del motivo por el que una combinación de fármacos se considera una interacción.
    \item Visualización de las enzimas implicadas en la interacción.
    \item Identificación del principio activo con mayor influencia sobre la interacción detectada.
    \item Presentación de los efectos adversos asociados.
    \item Propuesta de alternativas terapéuticas.
\end{itemize}

Asimismo, se realizó un estudio de las principales fuentes de información farmacológica disponibles con el objetivo de seleccionar aquellas que proporcionaban la información necesaria para el desarrollo del algoritmo y la construcción de la base de datos.

\textbf{Sprints asociados}: Esta etapa fue previa al inicio de la planificación mediante \textit{sprints}, por lo que no tiene ningún \textit{sprint} asociado.

\subsection*{Elaboración de la base de datos}

Durante esta etapa se definieron, junto con las tutoras, los criterios necesarios para el diseño de la base de datos y del algoritmo de análisis. Entre las principales decisiones adoptadas destacan:

\begin{itemize}
    \item Limitar la información a las enzimas del sistema CYP con mayor relevancia clínica.
    \item Establecer un criterio de prioridad para los principios activos que presentan más de una vía metabólica principal.
    \item Definir el procedimiento para la propuesta de alternativas terapéuticas consideradas seguras por la aplicación.
\end{itemize}

Asimismo, se diseñó la arquitectura general del sistema, estableciendo los distintos módulos que componen la aplicación y las relaciones existentes entre ellos. También se definió la estructura de la base de datos utilizada para almacenar la información farmacológica y el flujo de trabajo seguido por el algoritmo para generar las recomendaciones terapéuticas.

Finalmente, se diseñó la interfaz gráfica de usuario, priorizando la simplicidad, la claridad y la facilidad de interpretación de los resultados obtenidos.

\textbf{Sprints asociados}: Sprint 0, Semana 2, Semana 3 y Semana 4.

\subsection*{Desarrollo del algoritmo}

La fase de implementación comprendió el desarrollo progresivo de los diferentes componentes que integran la aplicación.

En primer lugar, se implementó el módulo encargado de obtener y procesar la información farmacológica procedente de las distintas fuentes de datos. Posteriormente, se desarrolló el algoritmo responsable de detectar las posibles interacciones farmacológicas, determinar su mecanismo de acción y generar las recomendaciones terapéuticas correspondientes.

\textbf{Sprints asociados}: Semana 5 y Semana 6.

\subsection*{Visualización, prueba y mejoras}

Una vez implementadas las funcionalidades principales, se realizaron diferentes pruebas utilizando casos reales de interacción farmacológica con el objetivo de verificar el correcto funcionamiento del algoritmo.

Estas pruebas permitieron comprobar la coherencia de las recomendaciones generadas, validar los resultados obtenidos y detectar posibles errores, que fueron corregidos de forma iterativa conforme avanzaba el desarrollo.

Posteriormente, se integró la interfaz gráfica con los distintos módulos implementados, obteniendo una aplicación completamente funcional.

Durante todo el desarrollo, el código fuente fue gestionado mediante Git y almacenado en un repositorio de GitHub, lo que permitió mantener un adecuado control de versiones y facilitar el seguimiento de la evolución del proyecto.

\textbf{Sprints asociados}: Semana 7, Semana 8 y Semana 9.

\subsection*{Redacción de la memoria}

De forma paralela a las últimas fases del desarrollo se descargaron los documentos y extensiones necesarias para la elaboración de la memoria del Trabajo Fin de Grado. Durante esta etapa se incorporaron las distintas secciones del documento y se realizaron revisiones periódicas junto con las tutoras, introduciendo las modificaciones necesarias para mejorar tanto el contenido como su presentación.

\textbf{Sprints asociados}: Semana 8, Semana 9 y \textit{Milestone} final.

\subsection*{Anexos y revisión final}

La fase final del proyecto estuvo dedicada a la elaboración de los anexos y a la revisión completa tanto de la memoria como de la aplicación desarrollada. Se incorporaron las últimas correcciones propuestas por las tutoras y se llevó a cabo una validación final de la interfaz y del funcionamiento general del sistema antes de la entrega del proyecto.

\textbf{Sprints asociados}: \textit{Milestone} final.

\section{Viabilidad legal}

El desarrollo del presente Trabajo Fin de Grado se ha realizado respetando la normativa vigente en materia de propiedad intelectual, protección de datos y licencias del software utilizado.

En primer lugar, la aplicación se ha desarrollado empleando herramientas y librerías de software libre, como Python y las bibliotecas necesarias para la implementación de la interfaz y del algoritmo de análisis. Todas ellas se distribuyen bajo licencias que permiten su utilización, modificación y distribución para fines académicos, respetando las condiciones establecidas por sus correspondientes autores.

El código fuente desarrollado durante el proyecto es de elaboración propia y se encuentra alojado en el repositorio público de GitHub 

(\texttt{https://github.com/CarmenRu/TFG}), lo que facilita su consulta y garantiza la trazabilidad del desarrollo mediante el historial de versiones.

En relación con la protección de datos, la aplicación no almacena ni procesa datos personales de pacientes o usuarios, ya que únicamente realiza el análisis de principios activos introducidos por el usuario. Por este motivo, durante el desarrollo del proyecto no ha sido necesario tratar información de carácter personal, evitando así la aplicación de medidas específicas relacionadas con el tratamiento de datos personales.

Por otra parte, la información farmacológica utilizada para construir la base de conocimiento procede de fuentes públicas y de acceso abierto, empleadas exclusivamente con fines académicos. En todo momento se han citado las fuentes consultadas en la memoria del trabajo, respetando los derechos de propiedad intelectual correspondientes.

Finalmente, la herramienta desarrollada tiene un propósito exclusivamente académico y de apoyo a la consulta de información farmacológica. En ningún caso pretende sustituir el criterio clínico de los profesionales sanitarios ni constituir una herramienta de diagnóstico o prescripción médica, por lo que los resultados obtenidos deben interpretarse únicamente como una ayuda para la toma de decisiones.
